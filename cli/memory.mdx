ZilMate's long-term memory system lets agents remember facts, preferences, and context across sessions. Everything you store is available not just to the CLI, but also to SDK sessions that share the same `ZILMATE_USER_ID`. Memories are retrieved by semantic search — you query with natural language and ZilMate surfaces the most relevant matches.

---

## Storage

By default, memories are stored locally under `.zilo-manager/` in your workspace directory. When Upstash Redis credentials are configured, ZilMate automatically uses Redis for cloud-backed, persistent storage that survives machine restarts and works across environments.

<Tip>
Memories stored via the CLI are accessible to SDK sessions using the same `ZILMATE_USER_ID`. Set that variable consistently across your environments to share a single memory store.
</Tip>

---

## Commands

### `zilmate remember "<text>"`

Store a fact or preference in long-term memory. This is a top-level command — run it from anywhere without a subcommand prefix.

```bash
zilmate remember "<text>"
```

Add optional tags to make memories easier to group and recall later:

```bash
zilmate remember "User prefers pnpm and App Router" --tag prefs --tag tooling
zilmate remember "Always format code responses with TypeScript, not JavaScript" --tag prefs --tag coding
zilmate remember "The staging server URL is https://staging.example.com" --tag infra --tag staging
zilmate remember "Use a warm but concise support tone" --tag prefs
```

**Flags:**

| Flag | Description |
|---|---|
| `-t, --tag <tag...>` | One or more tags to attach to this memory entry |

---

### `zilmate recall "<query>"`

Search long-term memory using natural language. ZilMate returns the top semantically matching entries. This is also a top-level command.

```bash
zilmate recall "<query>"
```

**Examples:**

```bash
zilmate recall "package manager preferences"
zilmate recall "staging environment details"
zilmate recall "how should I format code"
zilmate recall support --limit 5
```

**Flags:**

| Flag | Description |
|---|---|
| `-l, --limit <number>` | Maximum number of memories to return (default: `8`) |

---

### `zilmate memory list`

List all stored memories with their IDs, content, and tags. `zilmate memory` alone also shows this table.

```bash
zilmate memory list
```

---

### `zilmate forget <id>`

Delete a specific memory entry by its ID.

```bash
zilmate forget mem_abc123
```

To delete every memory at once, use the `--all` flag:

```bash
zilmate forget --all
```

<Warning>
`zilmate forget --all` permanently deletes every memory in your store. The CLI prompts for confirmation before proceeding.
</Warning>

---

## End-to-end example

```bash
# Store a few preferences and facts
zilmate remember "User prefers pnpm and App Router" --tag prefs --tag tooling
zilmate remember "Always respond in British English" --tag prefs --tag language
zilmate remember "Production database is PostgreSQL 16 on AWS RDS" --tag infra --tag db

# Review what is stored
zilmate memory list

# Recall by topic
zilmate recall "package manager preferences"
# → Returns: "User prefers pnpm and App Router"

# Remove a specific entry by ID
zilmate forget mem_abc123

# Clear everything (asks for confirmation)
zilmate forget --all
```

## SDK equivalent

The same memory store is accessible from the SDK:

```ts
import { createZilMate } from 'zilmate/server';

const zilmate = createZilMate({ sessionId: 'my-session' });

// Store a fact
await zilmate.remember({ text: 'User prefers pnpm and App Router', tags: ['prefs'] });

// Recall by semantic query
const hits = await zilmate.recall({ query: 'package manager', limit: 5 });
console.log(hits);
```

| CLI | SDK |
|---|---|
| `zilmate remember "<text>"` | `zilmate.remember({ text, tags })` |
| `zilmate recall "<query>"` | `zilmate.recall({ query, limit })` |
| `zilmate memory list` | `listMemories()` (internal) |
